# Dependency management

## Python

Python dependencies use a two-file workflow powered by [pip-tools](https://pip-tools.readthedocs.io/).

### Files

| File | Purpose | Edit? |
|---|---|---|
| `requirements.in` | Direct dependencies with `>=` lower bounds | Yes — this is the human-maintained file |
| `requirements.txt` | pip-compile output — fully pinned including all transitive deps | No — generated, do not edit by hand |

### Adding or upgrading a package

1. Edit `requirements.in` — add or update the `>=` bound
2. Recompile inside the dev container:
   ```bash
   make compile-deps
   ```
3. Reinstall into the venv:
   ```bash
   make setup-backend
   ```
4. Run tests to verify nothing broke:
   ```bash
   make check
   ```

### How `compile-deps` works

```bash
docker run --rm -v "$(CURDIR)":/workspace -w /workspace ccrag-dev \
  pip-compile requirements.in \
    --output-file requirements.txt \
    --strip-extras \
    --resolver=backtracking
```

- Runs inside the dev container to guarantee Python 3.11 resolution
- `--strip-extras` removes `package[extra]` annotations from output
- `--resolver=backtracking` uses pip's backtracking resolver for correct conflict detection
- The output annotates every transitive dependency with which direct dep requires it

### pip-tools location

`pip-tools` is installed globally in the dev container (`.devcontainer/Dockerfile`) and is **not** listed in `requirements.in`. It is a build-time tool only — it does not end up in the application image.

### Pinning philosophy

`requirements.in` uses `>=` lower bounds rather than exact pins. This means:

- pip-compile resolves the latest compatible set of all transitive deps
- `requirements.txt` contains exact `==` pins for full reproducibility
- Re-running `make compile-deps` on the same `requirements.in` may produce different `requirements.txt` pins as upstream packages release new versions — this is intentional and prompts a review

## Node

Node dependencies are managed with npm and a committed lockfile.

### Files

| File | Purpose | Edit? |
|---|---|---|
| `frontend/package.json` | Direct dependencies — all versions must be **exact** (no `^` or `~`) | Yes |
| `frontend/package-lock.json` | npm lockfile — full dependency tree with integrity hashes | No — generated by npm |

### Adding or upgrading a package

```bash
# Add a new package (inside dev container or make dev-shell)
cd frontend
npm install <package>@<exact-version>

# Then update package.json to remove any ^ or ~ that npm added
# and commit both package.json and package-lock.json
```

### Installing from lockfile

```bash
make setup-frontend   # runs npm ci — installs exact versions from package-lock.json
```

`npm ci` is used instead of `npm install` to ensure the lockfile is the source of truth. It will error if `package.json` and `package-lock.json` are out of sync.

### Pinning philosophy

All versions in `package.json` are exact (e.g. `"react": "18.3.1"`, not `"^18.3.1"`). This prevents unexpected upgrades when running `npm install` and makes the dependency graph fully auditable.

## Dependency review

The `dependency-reviewer` Claude agent audits both Python and Node dependencies for conflicts and CVEs before changes land in the repo. Run it via:

```
/agents dependency-reviewer
```

It will:
1. Run `pip install --dry-run` and `npm install --dry-run` to check for resolution conflicts
2. Run `npm audit` for known Node vulnerabilities
3. Web-search CVEs for production-facing packages
4. Propose specific fixes with reasons and ask for confirmation before editing any file

See `.claude/agents/dependency-reviewer.md` for the full agent specification.
